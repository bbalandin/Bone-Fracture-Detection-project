## О проекте

Данное приложение предназначено для автоматического обнаружения переломов костей на рентгеновских снимках. Система использует современные архитектуры YOLO (You Only Look Once) с поддержкой OBB (Oriented Bounding Boxes) для более точной локализации переломов сложной формы.

### Ключевые особенности:
- **6 классов переломов**: локоть, пальцы, предплечье, плечо, лопатка, запястье
- **Две модели**: быстрая (YOLO11n) и точная (YOLO11m)
- **Предобработка CLAHE**: улучшение контрастности для лучшей детекции
- **OBB формат**: ориентированные bounding boxes для сложных переломов

## Возможности

- Загрузка рентгеновских снимков (JPG, PNG)
- Выбор между быстрой и точной моделью
- Визуализация результатов с bounding boxes
- Отображение уверенности модели для каждого обнаруженного перелома
- Предобработка изображений с помощью CLAHE
- Поддержка OBB (Oriented Bounding Boxes)

### Ключевые выводы

1. **YOLO11n превзошла YOLO11m** по общей метрике mAP50 (0.233 и 0.195)
2. **Скорость**: YOLO11n работает в **7 раз быстрее** (196 мс и 1440 мс)
3. **Лучшие классы**:
   - YOLO11m: forearm fracture (0.321), wrist positive (0.278)
   - YOLO11n: humerus (0.449), forearm fracture (0.355)
4. **Проблемный класс**: elbow positive

### Ограничения и причины низких метрик

Лучший достигнутый результат map@50 = 0.22 на test и 0.35 на train. Основные причины низких метрик.

1. **Разнообразие ракурсов съёмки**. Рентгеновские снимки в датасете сделаны под разными углами, что создаёт значительную вариативность внешнего вида одного и того же типа перелома, что создаёт дополнительные сложности для модели.

2. **Неравномерное освещение**. Часть снимков имеет слишком слабое освещение, так например некоторые из снимков совсем тёмные, из-за была использована CLAHE — она дала прирост mAP на 5–10%.

3. **Малый размер изображений**. Разрешение снимков составляет около 512×512 пикселей, подробно про статистику по разрешениям показано в google colab. Это накладывает ограничения на детекцию мелких переломов (например, пальцев), где объект занимает небольшую область кадра.

4. **Нецелесообразность SAHI**. Метод Slicing Aided Hyper Inference, который обычно помогает на больших медицинских снимках, в данном случае не даёт преимущества: исходное разрешение уже небольшое, и нарезка на тайлы приводит к потере контекста без выигрыша в детализации.

5. **Дисбаланс классов**. Класс `wrist positive` представлен всего 6 изображениями, что делает обучение на нём нестабильным.

В будущем планирую написать несколько своих архитектур под эту задачу и найти ещё варианты для улучшения качества, например расширить датасет или использовать более качественные изображения

## Эксперименты

В рамках проекта было проведено множество экспериментов для поиска оптимальной архитектуры:

### Протестированные подходы

| Эксперимент | Описание | Результат |
|-------------|----------|-----------|
| **Faster R-CNN** | Двухэтапный детектор | Хуже чем YOLO по скорости и точности |
| **YOLO без OBB** | Конвертация в стандартные bounding boxes | Потеря точности на сложных переломах |
| **CLAHE vs без CLAHE** | Предобработка изображений | CLAHE улучшает mAP на 5-10% |
| **Аугментации YOLO** | Встроенные аугментации | Улучшение обобщающей способности |
| **Без аугментаций** | Чистые данные | Качество немного хуже чем с аугментациями |
| **Частичные аугментации** | Геометрические только | Компромиссный вариант |

**Подробный анализ** всех экспериментов представлен в Google Colab ноутбуках в папке `notebooks/`.

## Установка

### Локальная установка

```bash
# Клонируйте репозиторий
git clone https://github.com/bbalandin/bone-fracture-detection-project.git

# Создайте виртуальное окружение
python -m venv .venv

# Активируйте окружение
source .venv\Scripts\activate

# Установите зависимости
pip install -r requirements.txt

# можно отдельно запустить фронтенд и backend
# или можно сразу запустить один файл, который # использовался при деплое на Streamlit Cloud
streamlit run for_deploy.py
```
